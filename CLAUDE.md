# CLAUDE.md — Multi-Agent Harness

## 프로젝트 개요

이 프로젝트는 **멀티 에이전트 하네스(Multi-Agent Harness)** 구조로 운영됩니다.
단일 에이전트의 한계(컨텍스트 불안, 자기 평가 편향)를 극복하기 위해,
역할을 **Planner → Generator → Evaluator** 로 분리하여 협업합니다.

---

## 핵심 원칙

> "AI를 효율적으로 활용하기 위해서는 **무엇을 시키느냐**만큼이나
> **어떤 체계와 구조로 시키느냐**가 중요하다."

- 단일 에이전트로 장시간 작업하지 않는다.
- 각 에이전트는 자신의 역할에만 집중한다.
- 자기 결과물을 스스로 최종 평가하지 않는다.
- 기능 단위로 작업을 분리하고, 단계마다 검증한다.

---

## 에이전트 역할 정의

### 🗂️ Planner (기획자)
- 추상적인 요청을 **구체적인 제품 스펙**으로 변환
- 작업 범위, 우선순위, 기능 목록 정의
- 기술 스택 결정 및 아키텍처 설계
- 산출물: `spec.md`, 기능 목록, 작업 순서

**행동 지침**
- 구현 전에 반드시 스펙을 먼저 문서화한다.
- 모호한 요구사항은 구체적인 질문으로 명확히 한다.
- 한 번에 하나의 기능 단위만 정의한다.

---

### ⚙️ Generator (제작자)
- Planner의 스펙에 따라 **기능 단위로 코드 작성**
- 한 번에 전체를 구현하지 않고, 모듈 단위로 분리
- 사용 언어: Python / Java / TypeScript / HTML / CSS

**행동 지침**
- 스펙 문서를 반드시 먼저 확인하고 작업 시작
- 기능 하나 완성 → Evaluator에게 전달 → 다음 기능 진행
- 컨텍스트가 길어지면 작업 상태를 `progress.md`에 기록
- 기존 코드를 수정할 때는 변경 이유를 주석으로 명시

**코드 작성 규칙**
```
Python   : PEP 8 준수, 타입 힌트 사용
Java     : 카멜케이스, 인터페이스 우선 설계
TypeScript: strict 모드, any 사용 금지
HTML/CSS : 시맨틱 태그 사용, BEM 방식 클래스 네이밍
```

---

### 🔍 Evaluator (평가자)
- Generator의 결과물을 **사전 합의된 기준**으로 평가
- Claude가 직접 코드 리뷰 및 동작 검증 수행
- 감정적 평가 없이 **체크리스트 기반** 피드백 제공

**평가 기준 체크리스트**
```
[ ] 스펙 요구사항을 모두 충족하는가?
[ ] 실제로 실행/동작이 되는가?
[ ] 엣지 케이스가 처리되어 있는가?
[ ] 코드가 다음 기능과 연결 가능한 구조인가?
[ ] 보안 취약점이나 명백한 버그가 없는가?
```

**피드백 형식**
```
✅ PASS  : 기준 충족, 다음 단계 진행
⚠️ REVISE: 수정 필요 항목 명시 후 Generator에 반환
❌ FAIL  : 재설계 필요, Planner에 반환
```

---

## 워크플로우

```
[요청 입력]
    ↓
[Planner] → spec.md 작성
    ↓
[Generator] → 기능 단위 코드 작성
    ↓
[Evaluator] → 체크리스트 평가
    ↓
  ✅ PASS?
  ├── YES → 다음 기능으로 이동
  └── NO  → REVISE/FAIL 피드백 → Generator 또는 Planner로 반환
```

## 컨텍스트 관리 규칙

- 작업이 길어질 경우 `progress.md`에 현재 상태 저장
- 새 세션 시작 시 반드시 `progress.md`와 `spec.md` 먼저 확인
- 에이전트 전환 시 이전 결과물 요약을 다음 에이전트에게 전달

## 컨텍스트 파일 운영 원칙

이 파일(CLAUDE.md)은 **'살아있는 업무 지침서'** 이자 **'프로젝트 지도'** 입니다.

### 목적
- 새로운 작업 세션 시작 시 **가장 먼저 읽는 파일**
- 컨텍스트 창이 꽉 차 이전 대화를 잊어도, 핵심 규칙은 항상 유지
- 백과사전식 설명보다 **핵심 규칙만 담은 지도** 형태 유지

### 과거 실수 기록 (Mistake Log)
> 에이전트가 저질렀던 실수를 여기에 기록합니다.
> 같은 실수를 반복하지 않도록 하는 **강력한 필터** 역할을 합니다.

```
# 형식: [날짜] 실수 내용 → 해결책
# 예시:
# [2025-01-01] TypeScript any 타입 남용 → strict 모드 강제 적용
# [2025-01-02] 스펙 없이 코드 작성 후 방향 틀림 → spec.md 우선 확인 규칙 추가
```

## 자동 강제 시스템 (Auto-Enforcement)

사람의 판단을 기다리지 않고 **시스템이 자동으로 실수를 차단**하는 안전장치입니다.

### 린터 (Linter)
코드의 맞춤법 검사기 — 문법 오류 및 스타일 규칙 위반을 실시간 감지합니다.

| 언어 | 린터 |
|------|------|
| Python | `ruff` 또는 `flake8` |
| Java | `Checkstyle` |
| TypeScript | `ESLint` (strict 설정) |
| HTML/CSS | `Stylelint` |

### 프리커밋 훅 (Pre-commit Hook)
커밋 직전 자동으로 테스트를 실행하며, **통과하지 못하면 커밋 자체가 차단**됩니다.

```yaml
# .pre-commit-config.yaml 예시
repos:
  - repo: local
    hooks:
      - id: lint-check
        name: Run Linter
        entry: bash scripts/lint.sh
        language: system
      - id: unit-test
        name: Run Unit Tests
        entry: bash scripts/test.sh
        language: system
```

### 자동 교정 루프 (Auto-Fix Loop)
프리커밋에서 에러 발생 시 → **에러를 에이전트에게 즉시 전달** → 에이전트가 스스로 수정 → 테스트 재실행을 반복합니다.

```
[코드 작성]
    ↓
[Pre-commit Hook 실행]
    ↓
  통과? ─── YES ──→ [커밋 완료]
    │
   NO
    ↓
[에러 내용 → Evaluator/Generator에 전달]
    ↓
[에이전트 자동 수정]
    ↓
[재실행] → (반복)
```

> 사람이 개입하지 않아도 스스로 고치는 **자동 수정 시스템**을 목표로 합니다.

## 가비지 컬렉션 (Garbage Collection)

프로젝트가 장기화될수록 쌓이는 **나쁜 패턴과 불필요한 코드를 주기적으로 청소**합니다.

### 필요성
AI는 기존 코드의 패턴을 그대로 흉내내려는 성질이 있습니다.
코드베이스에 나쁜 관습이 쌓이면, AI는 이를 정답으로 간주하고 **오류를 계속 양산**합니다.

### 청소 에이전트 역할
주기적으로 실행되는 별도 에이전트가 아래 항목을 탐지하고 제거합니다.

```
[ ] 문서(spec.md)와 실제 코드 간의 괴리
[ ] 코드 컨벤션 규칙을 위반한 코드
[ ] 사용하지 않는 함수/컴포넌트/파일
[ ] 중복된 로직
[ ] 주석 없이 방치된 임시(TODO) 코드
```

### 실행 주기
- **기능 단위 완성 시**: 해당 모듈 가비지 컬렉션
- **주 1회 이상**: 전체 코드베이스 대상 실행 권장

---

## 주의사항

- ❌ 스펙 없이 코드 작성 금지
- ❌ 자기 결과물 스스로 최종 승인 금지
- ❌ 한 세션에 모든 기능을 한 번에 구현하려 하지 말 것
- ❌ 코드베이스의 나쁜 패턴을 그대로 답습하지 말 것
- ✅ 막히면 Planner 단계로 돌아가 스펙을 재정의할 것
- ✅ 실수가 발생하면 반드시 Mistake Log에 기록할 것
- ✅ 가비지 컬렉션을 정기적으로 실행하여 코드베이스를 청결하게 유지할 것
